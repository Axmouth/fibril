Fibril currently has Rust and TypeScript clients for the same core broker surface:

- connect with optional auth
- publish with or without confirmation
- delayed publish
- manual acknowledgements
- client-side auto-ack convenience
- message helpers for msgpack, JSON, raw bytes, and text payloads

The examples below are source-tree examples for the current pre-alpha API. They are useful for building against this repository, but not yet a stable package contract.

## Connect

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
use fibril_client::ClientOptions;

let client = ClientOptions::new()
    .auth("fibril", "fibril")
    .connect("127.0.0.1:9876")
    .await?;
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
import { ClientOptions } from "@fibril/client";

const client = await new ClientOptions()
  .withAuth("fibril", "fibril")
  .connect("127.0.0.1:9876");
```

  </TabItem>
</Tabs>

## Publish

Plain publish uses the common unconfirmed path. Confirmed publish waits for the broker to acknowledge the stored message and returns the topic offset. For pipelined confirmed publishing, send with a confirmation handle and await those handles later.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
let publisher = client.publisher("email.send")?;

publisher
    .publish("hello")
    .await?;

let offset = publisher
    .publish_confirmed("needs an offset")
    .await?;

let confirmation = publisher
    .publish_with_confirmation("pipeline me")
    .await?;
let pipelined_offset = confirmation.confirmed().await?;
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
const publisher = client.publisher("email.send");

await publisher.publish("hello");

const offset = await publisher.publishConfirmed("needs an offset");

const confirmation = await publisher.publishWithConfirmation("pipeline me");
const pipelinedOffset = await confirmation.confirmed();
```

  </TabItem>
</Tabs>

## Delayed Publish

Delayed publish uses a distinct protocol frame and stores a `not_before` Unix-millisecond deadline. It does not add delay bytes to the common publish frame.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
use std::time::Duration;

publisher
    .publish_delayed("send later", Duration::from_secs(30))
    .await?;

let offset = publisher
    .publish_delayed_confirmed("send later and return offset", 30u64)
    .await?;
```

Rust numeric delay helpers are seconds. `std::time::Duration` is also accepted.

  </TabItem>
  <TabItem label="TypeScript">

```ts
await publisher.publishDelayed("send later", 30_000);

const offset = await publisher.publishDelayedConfirmed(
  "send later and return offset",
  new Date(Date.now() + 30_000),
);
```

TypeScript numeric delays are milliseconds. Passing a `Date` uses that absolute deadline.

  </TabItem>
</Tabs>

## Message Payloads

Plain values are encoded as msgpack by default. Use explicit message helpers when you want JSON, raw bytes, text, or custom headers.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
use fibril_client::NewMessage;

publisher
    .publish(NewMessage::json(&serde_json::json!({
        "id": 42,
        "kind": "welcome",
    }))?)
    .await?;

publisher
    .publish(
        NewMessage::content("plain text")
            .header("x-trace-id", "abc123"),
    )
    .await?;

publisher
    .publish(NewMessage::raw(vec![1, 2, 3]))
    .await?;
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
import { NewMessage } from "@fibril/client";

await publisher.publish(
  NewMessage.json({
    id: 42,
    kind: "welcome",
  }),
);

await publisher.publish(
  NewMessage.content("plain text")
    .header("x-trace-id", "abc123"),
);

await publisher.publish(NewMessage.raw(new Uint8Array([1, 2, 3])));
```

  </TabItem>
</Tabs>

Fibril reserves `fibril.*` and `stroma.*` headers for system metadata. User code should not set headers with those prefixes. See [metadata policy](/latest/development/metadata-policy/) for the development note.

Dead-letter policy is configured through the admin UI/API rather than the Rust or TypeScript client APIs today. Client code should publish, consume, and settle messages; operators configure where exhausted messages go.

## Manual Acknowledgements

Manual acknowledgement is the primary processing model. Messages are leased while inflight. Consumers settle each message explicitly.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
let mut sub = client
    .subscribe("email.send")?
    .group("workers")?
    .prefetch(32)
    .sub_manual_ack()
    .await?;

while let Some(msg) = sub.recv().await {
    let body = msg.content()?;

    match send_email(body).await {
        Ok(()) => {
            msg.complete().await?;
        }
        Err(_) => {
            msg.retry().await?;
        }
    }
}
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
const sub = await client
  .subscribe("email.send")
  .group("workers")
  .prefetch(32)
  .subManualAck();

for await (const msg of sub) {
  const body = msg.content();

  try {
    await sendEmail(body);
    await msg.complete();
  } catch {
    await msg.retry();
  }
}
```

  </TabItem>
</Tabs>

`retry()` requeues immediately. Delayed retry is still partial and should not be treated as supported user-facing behavior yet.

## Dead Letter Workflow

Dead-letter routing is configured by an operator through the admin UI/API. Client code does not declare DLQ policy directly; it publishes and settles messages normally.

For example, configure a global DLQ target:

```http
PUT /admin/api/global-dlq
```

```json
{
  "expected_version": 0,
  "target": {
    "tp": "_dlq.email",
    "group": null
  }
}
```

Then configure a source queue to use that global target after retries are exhausted:

```http
PUT /admin/api/queue-dlq
```

```json
{
  "tp": "email.send",
  "group": "workers",
  "policy": "global",
  "target": null,
  "max_retries": 3
}
```

Application clients keep using normal publish/consume code:

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
let publisher = client.publisher_grouped("email.send", "workers")?;
publisher.publish_confirmed(NewMessage::json(&job)?).await?;

let mut sub = client
    .subscribe("email.send")?
    .group("workers")?
    .prefetch(32)
    .sub_manual_ack()
    .await?;

while let Some(msg) = sub.recv().await {
    if process(msg.deserialize()?).await.is_ok() {
        msg.complete().await?;
    } else {
        msg.retry().await?;
    }
}
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
const publisher = client.publisherGrouped("email.send", "workers");
await publisher.publishConfirmed(NewMessage.json(job));

const sub = await client
  .subscribe("email.send")
  .group("workers")
  .prefetch(32)
  .subManualAck();

for await (const msg of sub) {
  if (await process(msg.deserialize())) {
    await msg.complete();
  } else {
    await msg.retry();
  }
}
```

  </TabItem>
</Tabs>

Once the configured retry limit is exhausted, the broker routes the failed message to `_dlq.email`. Subscribe to that queue like any other queue when you want inspection or replay tooling.

## Decode Messages

`deserialize()` chooses a decoder from `content-type`. Missing content type defaults to msgpack.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
#[derive(serde::Deserialize)]
struct Job {
    id: u64,
}

let job: Job = msg.deserialize()?;
let raw: &[u8] = msg.raw();
let text: &str = msg.content()?;
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
type Job = {
  id: number;
};

const job = msg.deserialize<Job>();
const raw = msg.raw();
const text = msg.content();
```

  </TabItem>
</Tabs>

## Auto Ack

Auto-ack subscriptions are a convenience mode. The current client behavior tracks acknowledgement client-side; use manual ack when processing correctness matters.

<Tabs syncKey="client-language">
  <TabItem label="Rust">

```rust
let mut sub = client
    .subscribe("metrics")?
    .prefetch(128)
    .sub_auto_ack()
    .await?;

while let Some(msg) = sub.recv().await {
    observe(msg.deserialize::<Metric>()?);
}
```

  </TabItem>
  <TabItem label="TypeScript">

```ts
const sub = await client
  .subscribe("metrics")
  .prefetch(128)
  .subAutoAck();

for await (const msg of sub) {
  observe(msg.deserialize<Metric>());
}
```

  </TabItem>
</Tabs>
